## 🏗️ Overall Architecture

The core problem is that files, pipelines, and the engine live on three different machines.  
The solution is to make **Git the single source of truth**, and use **SSH to bridge the Raspberry Pi → VM**.

```
┌─────────────────┐     git push      ┌──────────────────────┐
│  Raspberry Pi   │ ─────────────────▶│     GitHub Repo      │
│  - pipelines    │                   │  - .aeon files       │
│  - ground truth │ ◀─────────────────│  - .rein files       │
│  - analysis     │     git pull      │  - results JSONs     │
└─────────────────┘                   │  - ground truth CSVs │
                                      └──────────────────────┘
                                                 │
                                            git pull
                                                 ▼
┌─────────────────────────────────────────────────────────────┐
│              Work Computer (Windows)                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │           VirtualBox VM (Linux)                     │   │
│  │  - RE:IN engine                                     │   │
│  │  - pulls .rein files, pushes result JSONs back      │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

## 🔌 Remote Access Strategy (for your partner)

The cleanest zero-config solution is **Tailscale**. It creates a private VPN between machines without needing to open router ports or configure anything complex.

**Setup steps (do this once):**


1. Install [Tailscale](https://tailscale.com) on the **Windows work computer** and on your **partner's computer**
2. Both join the same Tailscale network (free tier is enough)
3. Enable SSH on the VM inside VirtualBox: `sudo apt install openssh-server`
4. In VirtualBox → Settings → Network → Port Forwarding: add rule `Host 2222 → Guest 22`
5. Partner connects from anywhere with:
   ```bash
   ssh -p 2222 user@<tailscale-ip-of-work-computer>
   ```

## 🔬 Experiment 1: Sanity Check — *"Can RE:IN Do What It Claims?"*

### Goal
Verify the full pipeline works correctly end-to-end and that RE:IN recovers the correct network structure from synthetic observations on simple, manually verifiable models.

### Model Selection Agenda
Pick **5 small, clean models** from Biodivine:
- N ≤ 10 genes
- No input variables (every node has at least one incoming edge)
- Pure Boolean (no multi-valued logic)
- Models where you can manually verify the ground truth yourself

**Why this selection:** Small enough that if something is wrong in the pipeline, you will see exactly where. Clean enough to eliminate confounding factors.

### Selected Models (5 of 6 candidates)

Selected from `initial_screening_v2.csv` (E1_candidate label). All 6 candidates have 0 inputs, pure Boolean logic, and feedback loops. We dropped one duplicate topology to maximize diversity.

| # | Model ID | Name | N | Density | MaxInDeg | AvgReg | Selection Rationale |
|---|----------|------|---|---------|----------|--------|---------------------|
| 1 | 007_5_0 | CORTICAL-AREA-DEVELOPMENT | 5 | 0.70 | 4 | 2.8 | Smallest model, very dense (70% of possible edges exist). Classic neurodevelopment model — easy to manually trace all 14 edges on paper. |
| 2 | 088_6_0 | MIR-9-NEUROGENESIS | 6 | 0.37 | 2 | 1.83 | Moderate density, low max in-degree (2) — every node has at most 2 regulators. Tests RE:IN on a "flat" topology where no node is a hub. |
| 3 | 158_7_0 | LAMBDA-PHAGE-LYSOGENY | 7 | 0.71 | 6 | 4.29 | Highest average regulators of all candidates (4.29/node). One node has 6 incoming edges — a strong hub. Tests RE:IN under high connectivity. |
| 4 | 110_9_0 | ASYMMETRIC-CELL-DIVISION-B | 9 | 0.17 | 2 | 1.33 | Sparsest candidate (only 17% edge density). 9 nodes but very loosely connected. Tests if RE:IN works when there's little signal to constrain the search. |
| 5 | 031_9_0 | CELL-CYCLE-TRANSCRIPTION | 9 | 0.26 | 4 | 2.11 | 9 nodes, moderate density — fills the gap between sparse id-110 and dense id-158. Well-studied cell cycle model. |

**Dropped:** id-109 ASYMMETRIC-CELL-DIVISION-A (5 nodes, density=0.75) — nearly identical topology to id-007. Keeping both would waste a slot with no new information.

### Protocol

```
For each of the 5 models:
1. Run aeon_to_rein.py       → produces .rein file (all interactions set to Optional)
2. Run rein_add_experiments.py with K=10, N=5
3. Copy .rein file to VM (via git)
4. Run RE:IN engine on the VM
5. Pull result JSON back (via git)
6. Run analyzer.py           → compare output L-values vs. ground truth
```

### Fixed Parameters
| Parameter | Value |
|-----------|-------|
| K (steps per trajectory) | 10 |
| N (number of experiments) | 5 |
| Timeout | 60 seconds |

### Success Criteria

| Metric | ✅ Success | ❌ Failure |
|--------|-----------|-----------|
| Satisfiability | SAT for all 5 models | Any UNSAT result |
| L-value match | 100% match with ground truth | Any mismatch |
| Runtime | < 60 seconds per model | Timeout |
| Solution count | 1 or very few solutions | Explosion of solutions |

### What This Proves
The pipeline is correctly wired and RE:IN is functioning as a synthesis engine — not returning arbitrary outputs. This is the baseline from which all other experiments build.

## 🔬 Experiment 2: Stress Test — *"Where Does RE:IN Break?"*

### Goal
Characterize the computational limits of RE:IN as a function of network complexity. Produce a clear curve of **runtime vs. model size** and identify the threshold beyond which synthesis becomes infeasible.

### Model Selection Agenda
Select **20 models** stratified into 4 size bins:

| Bin | N (gene count) | Models | Selection note |
|-----|---------------|--------|----------------|
| A | 5–10 | 5 | Include 1 sparse + 1 dense |
| B | 11–20 | 5 | Include 1 sparse + 1 dense |
| C | 21–30 | 5 | Include 1 sparse + 1 dense |
| D | 31–40 | 5 | Include 1 sparse + 1 dense |

Within each bin, vary **edge density** (sparse vs. dense). This lets you separate the effect of size vs. connectivity on runtime.

**Why this selection:** A stratified sample gives you a statistically meaningful performance curve. Without stratification you can't tell whether slowdowns are caused by size or by specific network topology.

## 🔬 Experiment 3: Observation Sufficiency — *"How Much Data Does RE:IN Need?"*

### Goal
Find the **minimum number of synthetic observations** (varying K and N) required for RE:IN to reliably reconstruct the original network. This is the most scientifically original experiment and directly supports the claim that "sufficient experimental observations" are needed for accurate synthesis.

### Model Selection Agenda
Pick **1 medium model** as a fixed reference:
- N ≈ 15–20 genes
- Clean, SAT-able (confirmed in Experiment 1 or 2)
- Has a mix of activators and inhibitors (not trivially simple)

**Why one model:** The model is held constant — the only variable is observation quantity. This gives a clean, interpretable result with no confounding factors.

### Selected Model: id-057 FANCONI-ANEMIA-AND-CHECKPOINT-RECOVERY

**File:** `057_15_0.aeon` — 15 genes, 0 inputs, density = 0.29, maxInDeg = 8, avgReg = 4.07

Selected from 5 E3_candidate models. All candidates had 15–20 nodes, 0 inputs, Boolean logic, and feedback loops. Here is the full comparison:

| Candidate | N | Density | Activations | Inhibitions | Mix (% inh.) | Self-loops | Verdict |
|-----------|---|---------|-------------|-------------|--------------|------------|---------|
| id-026 BUDDING-YEAST-CELL-CYCLE-2009 | 18 | 0.19 | 38 | 20 | 34% | 8 | Good but many self-loops inflate recovery |
| id-055 HUMAN-GONADAL-SEX-DETERMINATION | 19 | 0.20 | 48 | 20 | 29% | 3 | Skewed toward activation |
| **id-057 FANCONI-ANEMIA-CHECKPOINT** | **15** | **0.29** | **38** | **23** | **37%** | **2** | **Winner** |
| id-074 T-LGL-SURVIVAL-NETWORK-2011-REDUCED | 18 | 0.14 | 14 | 29 | 67% | 2 | Heavily skewed to inhibition |
| id-208 HEMATOPOIESIS-AGING | 15 | 0.17 | 25 | 11 | 30% | 3 | Too sparse, low complexity |

**Why id-057 is the best choice:**

1. **Best activator/inhibitor balance** — 38 activations vs 23 inhibitions (37% inhibitory). Closest to an even mix, meaning RE:IN must correctly distinguish both edge types. Models skewed to one sign (like id-074 at 67% inhibitory) are easier to guess.
2. **Highest density (0.29)** — Most edges per node of all candidates. Gives RE:IN the most constraints to work with, producing a cleaner gradient from "not enough data" to "sufficient data" across the K x N grid.
3. **Highest avg regulators (4.07)** — Each gene is regulated by ~4 others on average. Large search space per gene means varying K and N will have a visible effect on recovery ratio.
4. **Minimal self-loops (only 2)** — Self-loops add trivial constraints (a gene regulating itself is always recoverable). Only 2 means the experiment measures real reconstruction difficulty, not self-loop padding.
5. **15 genes** — Right in the middle of the 15–20 target range. Small enough that RE:IN should solve it, large enough that the K x N grid will show meaningful variation.
6. **DNA repair pathway** — Biologically well-characterized, which helps with biological interpretation in the project book.

### Protocol

Run a **full grid** of (K, N) combinations on the single reference model:

| | N=1 | N=3 | N=5 | N=10 |
|-|-----|-----|-----|------|
| **K=5** | run | run | run | run |
| **K=10** | run | run | run | run |
| **K=15** | run | run | run | run |

**12 runs total.** For each run record:
- SAT / UNSAT
- Recovery ratio (% of genes with correct L-value)
- Stability ratio (% of genes with variance = 0)
- Number of concrete solutions found

### Success Criteria

| Expected Finding | Meaning |
|-----------------|---------|
| Low K + Low N → many solutions, low recovery | Not enough data to constrain the search |
| Increasing K or N → fewer solutions, higher recovery | Observations are doing their job |
| A plateau exists at some (K*, N*) where recovery ≈ 100% | You've found the sufficiency threshold |

### Output for Project Book
A **heatmap** (K × N) with recovery ratio as the cell value. One clear recommendation statement:

> *"For a model of approximately N=15 genes, a minimum of K=10 steps and N=5 experiments is sufficient for reliable reconstruction."*

### What This Proves
Quantifies the data requirements of RE:IN — a directly useful contribution for any researcher wanting to apply this tool to a new biological network. Bridges computational output to practical experimental design guidance.

## 📋 VM Access Guide (draft — expand into `VM_ACCESS_GUIDE.md`)

```markdown
## Prerequisites
- Tailscale installed on your machine
- You've been added to the project's Tailscale network (ask Ofri)
- Git installed locally

## Step 1: Connect to the VM
ssh -p 2222 <username>@<tailscale-ip>

## Step 2: Pull latest files from GitHub
cd ~/rein-workspace
git pull origin main

## Step 3: Run the RE:IN engine on a model
./run_engine.sh results/experiment_X/model_name.rein

## Step 4: Push results back
git add results/
git commit -m "run: experiment X model_name"
git push origin main

## Troubleshooting
- SSH timeout      → check Tailscale is running on both machines
- Engine hangs     → timeout is 300s, it will kill automatically
- UNSAT result     → check that .rein file has self-loops for input nodes
- Git push fails   → do git pull first, then push again
```
